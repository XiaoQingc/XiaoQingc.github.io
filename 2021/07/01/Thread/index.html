多线程笔记

多线程
Java 多线程
发布日期:   2021-07-01

一 线程的创建

1.继承Thread类
*通过继承Thread类，来具备多线程的能力
*启动线程：子类对象.start（）；
*不建议使用：避免OOP（面向对象）单继承局限性

2.实现Runnable接口
*通过实现Runnable接口来具备多线程的能力
*启动线程：new Thread对象+传入目标对象.start()
*推荐使用：避免单继承的局限性，灵活方便，方便同一个对象被多个线程使用。
3.实现callable接口
*callable 可以定义返回值
*可以抛出异常

*程序、进程、线程的定义和区别
定义:
*程序是实现某个事件而用语言编写的指令序列集合
*进程是运行着的程序
*线程是比进程更小的执行单位,他是进程中的一条执行路径

​ 区别:
​ *不同的进程,占用的独立存储空间是不一样的. 而线程没有独立的存储空间,而是和所属进程的其他线程共享一个存储空间

二 线程的运用

1.线程停止

通过设置标识位来进行线程停止,当flag=false时线程停止.

2.线程休眠

Thread.sleep()

  • sleep(时间) 指定当前线程阻塞的毫秒数*

  • sleep存在异常InterruptedException

  • sleep时间达到后线程进入就绪状态*

  • sleep可以模拟网络延时,倒计时等;

每一个对象都有一个锁,sleep不会释放锁;

3.线程礼让

Thread.yield()

  • 礼让线程,让当前正在执行的线程暂停,但不阻塞
  • 将线程从运行状态转为就绪状态
  • 让cpu重新调度,礼让不一定成功!看cpu心情

4.线程强制执行

​ 需要强插的线程对象.join()

  • join合并线程,待此线程执行完成后,在执行其他线程,其他线程阻塞
  • 可以想象成插队

5.线程状态

  • new
    • Thread t = newThread()
    • 线程对象一旦创建就进入到了新生状态
  • 就绪状态
    • 当调用start()方法,线程立即进入就绪状态,但不意味着立即调度执行
  • 运行状态
    • 进入运行状态,线程才真正执行线程体的代码块
  • 阻塞状态
    • 当调用sleep(),wait()或同步锁定时,线程进入阻塞状态,就是代码不往下执行,阻塞时间解除后,重新进入就绪状态,等待cpu调度执行
  • dead
    • 线程中断或者结束,一旦进入死亡状态,就不能再次启动
  • *观测线程状态(state)
    • 线程对象.state()
      观测当前线程使用状态

6.线程的优先级

  • setpriority()设置线程优先级方法
  • getpriority()获取当前线程的优先级
  • Thread.MIN.PRIORITY(最小优先级) 优先级 = 1
  • Thread.MAX.PRIORITY(最大优先级) 优先级 = 10
  • Thread.NORM.PRIORITY(默认优先级) 优先级 = 5
  • 设置范围为(1-10)
  • 优先级高低,只是cpu调用的概率高低.优先级低不代表不会先调用,这都是看cpu的调度

7.守护线程

  • 线程分为用户线程和守护线程
  • 虚拟机必须保证用户线程执行完毕
  • 虚拟机不用等在守护线程执行完毕
  • 如:后台记录操作日志,监控内存,垃圾回收等待..

8.线程同步

*由于我们可以通过 private 关键字来保证数据对象只能被方法访问,所以我们只需要针对方法提出一套机制,这套机制就是synchronized 关键字,它包括两种方法 : synchronized 方法和 synchronized 方法块.

*同步方法 : public synchronized void method(int args){}
*synchronized方法控制”对象”的访问,没改对象对应一把锁,每个synchronized方法都必须获得该方法的对象的锁才能执行否则线程会 阻塞,方法一旦执行,就独占该锁,直到该方法返回才释放锁,后面被阻塞的线程才能获得这个锁,继续执行.

*缺陷 : 若将一个大的方法申明为synchronized 将会影响效率

*同步块 : synchronized(obj){}
*Obj称之为 同步监视器

  • obj 可以是任何对象,但是推荐使用共享资源作为同步监视器
  • 同步方法中无需指定同步监视器,因为同步方法的同步监视器就是this,就是当前对象本身,或者是class

*同步监视器的执行过长过程

  • 第一个线程访问,锁定同步监视器,执行其中的代码,
  • 第二个线程访问,发现同步监视器被锁定,无法访问
  • 第一个线程访问完毕,解锁同步监视器
  • 第二个线程访问,发现同步监视器解锁,然后锁定并且访问

9.死锁

*多个线程各自占有一些共享资源,并且互相等待其他线程占有的资源才能运行,而导致两个或者多个线程都在等下对方释放资源,都停止执行的情形,某一个同步块同时拥有两个或者以上的对象锁,就可能发现”死锁”问题. (也就是说当两个线程或者多个时,获取了共享资源,每个线程获取下一个资源都需要等对方资源释放完毕,但是每个线程需要获取对方的资源后才能释放资源,这就形成了死锁)

*产出死锁的四个条件:

  • 互斥条件 : 一个资源只能被一个进程使用

  • 请求与保持条件 : 一个进程因请求资源而阻塞时,对已获得的资源保持不放

  • 不剥夺条件 : 进程已获得的资源,在末使用之前,不能强行剥夺

  • 循环等待条件 : 若干进程之间形成一种头尾相接的循环等待资源关系

    避免其中任意一个或者多个条件时,就可以避免死锁发生

10.Look(锁)

*从JDK5.0开始,java提供了更加强大的线程同步机制—通过显示定义同步锁对象来实现同步.同步锁相当于是使用Lock对象充当.
*java.util.concurrent.Lock接口是控制多个线程对共享资源进行访问的工具.锁提供了对共享资源的独占访问,每次只能有一个线程对lock对象进行加锁,线程开始访问共享资源之前应该先获取lock对象.
*ReentrantLock类实现了Lock,它拥有与synchronized相同的并发性和内存语义,在实现线程安全的控制中,比较常用的是ReentrantLock,可以显式加锁丶释放锁.

  • Lock是显式锁 (手动开启和关闭锁,别忘记关闭锁)synchronized是隐式锁,出了作用域自动释放.
  • Lock只有代码块锁,synchronized有代码块锁和方法锁
  • 使用Lock锁,JVM将花费较少的时间来调度线程,性能更好.并且具有更好的扩展性(提供更多的子类)
  • 优先使用顺序:
         Lock>同步代码块(已经进入了方法体,分配了相应的资源)>同步方法(在方法体之外)

11.线程池

*ExecutorService : 真正的线程池接口.常见子类TreadPoolExecutor

  • void execute(Runnable command) : 执行任务/命令,没有返回值,一般用来执行Runnable
  • Futuresunbmit(Callabletask) : 执行任务,有返回值,一般用来执行Callble
  • void shutdown() : 关闭线程池

*Executors : 工具类丶线程池的工厂类,用于创建并返回不同类型的线程池

12.生产者消费者问题

*对于生产者,没有生产产品之前,通知消费者等待,生产产品后通知消费者马上来消费.
*对于消费者,在消费后,要通知生产者已经结束消费,需要生产新的产品以供消费.
*在生产者消费者问题中,仅有synchronized是不够的:

  • synchronized可组组长并发更新同一个共享资源,实现了同步
  • synchronized不能用来实现不同线程之间的消息传递(通信)

wait() 标识线程一直等待,直到其他线程通知,与sleep不同,会释放锁
notify()唤醒一个处于等待状态的线程
notifyAll()唤醒同一个对象所以调用wait方法的线程,优先级别高的线程优先调度
